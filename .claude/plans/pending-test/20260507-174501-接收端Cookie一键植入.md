## 一、需求摘要

**目标**：让用户在自己的"账号管理控制台"（网页 A）点一下按钮，新 tab 打开任意网站（如 x.com / instagram.com），自动植入预存的 cookie 完成登录。

**用户画像**：自用 / 小团队几人，无后端，无用户系统。

**关键交付物**：
1. 一个独立的油猴脚本 `universal-login-receiver.user.js`
2. 一份给"网页 A 的 AI"看的协议文档
3. 一个本地测试 HTML 用于自验证

**不在范围内**：
- 网页 A 本身（用户自己实现，按协议接入）
- localStorage / sessionStorage / IndexedDB 同步
- 多账号管理 / 后端系统
- 跨设备同步

## 二、架构决策（已与用户对齐）

| 维度 | 决策 | 取舍 |
|------|------|------|
| 通信通道 | URL hash `#__ulinject=` | 跨域可读、不进 server 日志、不需要 server |
| 触发方式 | A 站 `window.open(target + #__ulinject=...)` | 一次性、利用浏览器原生跳转 |
| 接收时机 | `@run-at document-start` | 抢在首屏请求前注入，避免 SPA 旧态污染 |
| 脚本拆分 | 新建独立脚本，不动现有 1.3.1 | 接收端职责独立，原 Cookie Manager 面板保留 |
| 注入范围 | **仅 cookie** | 用户选择最简方案，覆盖大多数登录场景 |
| 注入前清场 | **是**（复用原脚本清场逻辑思路） | 防止新旧账号串台 |
| 注入后行为 | 1s toast → `location.reload()` | 视觉反馈 + 清干净状态 |
| Cookie 格式 | **自动解析**（字符串 / JSON 都吃） | A 站 AI 自由选择 |
| 调试日志 | **保留** `console.log` | 自用为主，便于排查 |

### 关于安全 —— 用户已知情选择最弱模型

用户在采访中明确选择关闭所有自动防护：
- ❌ 不验证 HMAC 签名
- ❌ 不验证 referrer 来源
- ❌ 不校验 nonce 防重放
- ❌ 不校验 exp 过期
- ❌ 不弹用户确认框
- ❌ 不限制路径必须为根

**已知风险**：任何第三方网站只要放一个含 `#__ulinject=...` 的链接，用户点击即被静默植入攻击者 cookie，构成会话固定攻击（Session Fixation）。

**用户的对应策略**：自我约束 —— 不点不明链接。这是用户的主动决定。

**实施要求**：在 `docs/integration/A站接入指南.md` 顶部用醒目警告框说明此风险，让未来的协作者知情；在脚本注释里也写一份。

## 三、协议规范（A 站要遵守的）

### 3.1 URL 格式

```
https://<目标域名>/<任意路径>#__ulinject=<base64url(JSON payload)>
```

### 3.2 Payload 数据结构（最小版）

```json
{
  "v": 1,
  "cookies": "auth_token=abc; ct0=xyz; twid=u%3D123"
}
```

或结构化数组形式：

```json
{
  "v": 1,
  "cookies": [
    { "name": "auth_token", "value": "abc", "domain": ".x.com" },
    { "name": "ct0", "value": "xyz" }
  ]
}
```

### 3.3 编码

```
encoded = base64url(JSON.stringify(payload))
```

`base64url` = 标准 base64 但 `+` → `-`、`/` → `_`、去掉尾部 `=`。

### 3.4 A 站参考代码（接入指南里给出完整版）

```js
function injectLogin(targetSite, cookieString) {
  const payload = { v: 1, cookies: cookieString };
  const encoded = btoa(JSON.stringify(payload))
    .replace(/\+/g, '-').replace(/\//g, '_').replace(/=+$/, '');
  window.open(`https://${targetSite}/#__ulinject=${encoded}`, '_blank');
}
```

## 四、接收端流程设计

```
document-start 触发
    ↓
有 #__ulinject= ?
    ├─ 否 → 啥都不做
    └─ 是 ↓
        解析 base64 → JSON → payload
            ↓
        解析失败? → console.log 错误并退出
        解析成功? ↓
            ↓
        history.replaceState 清掉 hash
            ↓
        showToast("正在为你登录...")
            ↓
        await clearAllCookies()
        await clearStorage()
        await clearCacheStorage()
            ↓
        await injectCookies(payload.cookies)
            ↓
        setTimeout(() => location.reload(), 1000)
```

## 七、风险与回滚

| 风险 | 缓解 |
|------|------|
| 大站（X / IG）反爬识别异常注入 | 保留全套 GM_cookie 多 URL 策略；失败时日志可见 |
| URL hash 长度超限 | A 站只放精简 cookie；超过 6KB 时考虑 gzip+base64（暂不实现） |
| 接收脚本误执行（误判 hash） | hash 必须严格匹配 `#__ulinject=` 才解析 |
| 现有 user.js 受影响 | 新脚本完全独立、不改原文件，最坏情况只需禁用新脚本 |

回滚：删除 / 禁用 `universal-login-receiver.user.js` 即可。
